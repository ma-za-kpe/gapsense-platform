# Requirements Document

## Introduction

Phase 2 of the GapSense improvement plan replaces the brute-force curriculum graph building in the image analysis pipeline with hybrid vector search and graph traversal. The current `_build_curriculum_graph` method in `ImageAnalysisOrchestrator` dumps up to 100 curriculum nodes as raw JSON into the AI prompt, consuming 15,000–25,000 tokens of mostly irrelevant context. This causes the model to hallucinate curriculum codes, miss prerequisite relationships, and waste tokens on grade-irrelevant nodes. Phase 2 introduces pgvector-backed indicator-level embeddings, a lightweight image description step for query building, cosine similarity search (top_k=15), and a recursive CTE prerequisite walk (depth=2) to inject only 10–20 semantically relevant nodes (~2,500 tokens) into the prompt. Graceful fallback restores pre-Phase-2 behaviour when embeddings have not been generated or pgvector is unavailable. Phase 1 (infrastructure hardening) is a prerequisite and assumed complete. This phase does not touch OCR/Mathpix (Phase 3) or grade normalisation (Phase 4).

## Glossary

- **Orchestrator**: The `ImageAnalysisOrchestrator` class that owns the image analysis pipeline, including the `_build_curriculum_graph` method being replaced.
- **Embedding_Service**: A new service class (`EmbeddingService`) that generates vector embeddings from text using either OpenAI text-embedding-3-small (1536 dimensions) in production or all-MiniLM-L6-v2 (384 dimensions) in development.
- **Embedding_Job**: A standalone async script (`embedding_job.py`) that runs at curriculum import time to generate and store embeddings for all curriculum indicators in a given country and subject.
- **Indicator_Chunk**: The deterministic text string built per `CurriculumIndicator` for embedding, formatted as: `"Curriculum node: {node_code} — {node_title}\nIndicator: {indicator_code} — {indicator_title}\nCommon errors: {ep1}; {ep2}; {ep3}"`.
- **Vector_Search**: A cosine similarity query against the `embedding` column on `curriculum_indicators` using pgvector, returning the top_k most semantically similar indicators to a query vector.
- **Prerequisite_Walk**: A recursive CTE query that traverses the `curriculum_prerequisites` table upward from seed nodes to discover transitive prerequisite nodes up to a configurable depth.
- **Seed_Nodes**: The set of `CurriculumNode` IDs whose indicators were returned by Vector_Search — the starting points for the Prerequisite_Walk.
- **Query_Text**: A brief natural-language description of the mathematical content visible in the student image, generated by a lightweight AI call (Claude Haiku) and used as the input for embedding and Vector_Search.
- **Retrieval_Metadata**: A dictionary stored on `ImageAnalysisContext` that records which nodes were injected via Vector_Search vs Prerequisite_Walk, the total node count, and a preview of the Query_Text.
- **IVFFlat_Index**: An approximate nearest-neighbour index type provided by pgvector, configured with `lists=100` and `vector_cosine_ops` for cosine distance queries.
- **CurriculumPrerequisite**: The existing ORM model (`curriculum_prerequisites` table) with `source_node_id` (the dependent node) and `target_node_id` (the prerequisite node), used by the Prerequisite_Walk.
- **Fallback_Mode**: The degraded retrieval path activated when embeddings do not exist or pgvector is unavailable, which restores pre-Phase-2 behaviour by selecting curriculum nodes ordered by code with a row limit.

## Requirements

### Requirement 1: pgvector Extension and Embedding Column Migration

**User Story:** As a platform engineer, I want the pgvector extension enabled and an embedding column added to the `curriculum_indicators` table, so that indicator-level vector embeddings can be stored and queried using cosine similarity.

#### Acceptance Criteria

1. THE Alembic migration SHALL execute `CREATE EXTENSION IF NOT EXISTS vector` to enable pgvector.
2. THE Alembic migration SHALL add an `embedding` column of type `vector(1536)` to the `curriculum_indicators` table, with `nullable=True`.
3. THE Alembic migration SHALL add an `embedding_model` column of type `String(50)` to the `curriculum_indicators` table, with `nullable=True`, to record which model generated the embedding.
4. WHEN the migration runs on a database that already has the `vector` extension, THE migration SHALL succeed without error.
5. THE `CurriculumIndicator` ORM model SHALL include the `embedding` field mapped to `Vector(1536)` and the `embedding_model` field mapped to `String(50)`, both nullable.
6. THE migration downgrade SHALL drop the `embedding` and `embedding_model` columns and SHALL NOT drop the pgvector extension (other tables may use pgvector).

### Requirement 2: IVFFlat Index Creation with Minimum Vector Guard

**User Story:** As a platform engineer, I want an IVFFlat index on the embedding column that is only created when sufficient vectors exist, so that cosine similarity queries are fast without triggering pgvector errors on undersized datasets.

#### Acceptance Criteria

1. THE Embedding_Job SHALL create an IVFFlat index on `curriculum_indicators.embedding` using `vector_cosine_ops` with `lists=100` after embedding a country/subject batch.
2. IF fewer than 100 indicators have non-null embeddings for the country/subject, THEN THE Embedding_Job SHALL skip index creation and log a warning stating the indicator count and the 100-vector minimum.
3. WHEN the IVFFlat index already exists, THE Embedding_Job SHALL skip index creation without error.
4. THE IVFFlat index SHALL be named `idx_curriculum_indicators_embedding` for consistent reference.

### Requirement 3: EmbeddingService Implementation

**User Story:** As a developer, I want an EmbeddingService that generates vector embeddings from text using a configurable model backend, so that curriculum indicators and query text can be embedded consistently for semantic search.

#### Acceptance Criteria

1. THE Embedding_Service SHALL support two backends: OpenAI `text-embedding-3-small` (1536 dimensions) selected via `EMBEDDING_MODEL=openai`, and sentence-transformers `all-MiniLM-L6-v2` (384 dimensions) selected via `EMBEDDING_MODEL=minilm`.
2. THE Embedding_Service SHALL fix the backend at construction time based on the application settings and SHALL NOT switch backends during its lifetime.
3. THE Embedding_Service SHALL expose an `embed(text: str) -> list[float]` method that returns a single embedding vector.
4. THE Embedding_Service SHALL expose an `embed_batch(texts: list[str]) -> list[list[float]]` method that embeds multiple texts efficiently, batching OpenAI calls in groups of 100.
5. THE Embedding_Service SHALL expose a `build_indicator_chunk` method that accepts `node_code`, `node_title`, `indicator_code`, `indicator_title`, and `error_patterns: list[str]` and returns the deterministic Indicator_Chunk string.
6. WHEN the OpenAI API returns a rate-limit error, THE Embedding_Service SHALL retry with exponential backoff up to 3 attempts before raising the error.
7. IF the OpenAI API key is not configured and the backend is set to `openai`, THEN THE Embedding_Service SHALL raise a configuration error at construction time with a descriptive message.
8. THE Embedding_Service SHALL return vectors whose dimensionality matches the configured backend (1536 for OpenAI, 384 for MiniLM) for every call.

### Requirement 4: Curriculum Embedding Job

**User Story:** As a platform engineer, I want a standalone embedding job that generates and stores embeddings for all curriculum indicators in a given country and subject, so that Vector_Search has pre-computed embeddings to query against.

#### Acceptance Criteria

1. THE Embedding_Job SHALL accept `country`, `subject`, and an optional `force_refresh` flag as parameters.
2. THE Embedding_Job SHALL query all `CurriculumIndicator` rows for the specified country and subject where `embedding IS NULL`, or all rows when `force_refresh` is `True`.
3. THE Embedding_Job SHALL build Indicator_Chunk text for each indicator using `Embedding_Service.build_indicator_chunk`, loading error patterns from the `IndicatorErrorPattern` relationship.
4. THE Embedding_Job SHALL call `Embedding_Service.embed_batch` to generate embeddings and write the resulting vectors back to the `embedding` column and the model name to the `embedding_model` column on each `CurriculumIndicator` row.
5. THE Embedding_Job SHALL be idempotent — running the job multiple times without `force_refresh` SHALL skip indicators that already have embeddings.
6. WHEN `force_refresh` is `True`, THE Embedding_Job SHALL overwrite existing embeddings for the specified country and subject.
7. THE Embedding_Job SHALL return an `EmbeddingJobResult` dataclass containing: `country`, `subject`, `total_indicators`, `newly_embedded`, `already_embedded`, `errors`, and `duration_seconds`.
8. THE Embedding_Job SHALL validate that all existing embeddings for the country/subject use the same `embedding_model` as the current Embedding_Service backend before proceeding. IF a model mismatch is detected, THEN THE Embedding_Job SHALL abort with an error message instructing the operator to use `force_refresh`.
9. THE Embedding_Job SHALL provide a CLI entry point invocable as `python -m gapsense.jobs.embedding_job --country <country> --subject <subject> [--force-refresh]`.

### Requirement 5: Query Text Generation from Student Image

**User Story:** As a developer, I want the Orchestrator to generate a brief natural-language description of the mathematical content visible in a student image, so that the description can be embedded and used as the query vector for Vector_Search.

#### Acceptance Criteria

1. THE Orchestrator SHALL include a `_build_query_text` method that sends the student image to Claude Haiku with a prompt requesting a 2–3 sentence description of the mathematical topics and operations visible.
2. THE `_build_query_text` prompt SHALL instruct the model to name specific operations, number ranges, and visible error patterns without diagnosing the student.
3. THE `_build_query_text` prompt SHALL include the student grade and country for context.
4. IF the AI call in `_build_query_text` fails or returns an empty response, THEN THE Orchestrator SHALL fall back to a generic query string of `"{subject} {student_grade}"` and log a warning.
5. THE Orchestrator SHALL store the generated Query_Text on `ImageAnalysisContext.image_description` for downstream use and audit logging.

### Requirement 6: Vector Search for Semantically Relevant Indicators

**User Story:** As a developer, I want the Orchestrator to perform cosine similarity search against embedded curriculum indicators, so that only the most semantically relevant nodes are injected into the AI prompt.

#### Acceptance Criteria

1. THE Orchestrator SHALL include a `_vector_search` method that accepts a query vector, country, subject, and `top_k` parameter (default 15).
2. THE `_vector_search` method SHALL query `CurriculumIndicator` rows joined to `CurriculumNode`, filtered by country and subject, where `embedding IS NOT NULL`, ordered by cosine distance ascending, limited to `top_k` results.
3. THE `_vector_search` method SHALL return a list of `CurriculumIndicator` instances with their associated `CurriculumNode` relationships loaded.
4. IF `_vector_search` returns zero results (no embeddings exist for the country/subject), THEN THE Orchestrator SHALL fall back to selecting the first 20 indicators ordered by `CurriculumNode.code` for the country and subject, and log a warning indicating the embedding job has not run.
5. IF pgvector raises an `OperationalError` (extension not installed), THEN THE Orchestrator SHALL catch the error, fall back to the code-ordered SELECT with limit 20, and log an error.

### Requirement 7: Prerequisite Graph Walk via Recursive CTE

**User Story:** As a developer, I want the Orchestrator to walk the prerequisite graph upward from seed nodes, so that root-cause prerequisite nodes are included in the prompt context even when they are not direct semantic matches.

#### Acceptance Criteria

1. THE Orchestrator SHALL include a `_walk_prerequisites` method that accepts a set of Seed_Node IDs, a country, and a `depth` parameter (default 2).
2. THE `_walk_prerequisites` method SHALL execute a recursive CTE against the `curriculum_prerequisites` table, walking from `source_node_id` (dependent) to `target_node_id` (prerequisite) up to the specified depth.
3. THE `_walk_prerequisites` method SHALL return only the discovered prerequisite node IDs, excluding the Seed_Nodes themselves.
4. WHEN the Seed_Node set is empty, THE `_walk_prerequisites` method SHALL return an empty set without executing a database query.
5. IF the `curriculum_prerequisites` table contains no edges for the given country, THEN THE `_walk_prerequisites` method SHALL return an empty set and log a warning indicating prerequisite edges are empty.

### Requirement 8: Hybrid _build_curriculum_graph Replacement

**User Story:** As a developer, I want the `_build_curriculum_graph` method replaced with a hybrid retrieval pipeline, so that the AI prompt receives only 10–20 semantically relevant nodes with explicit prerequisite chains instead of 100 unfiltered nodes.

#### Acceptance Criteria

1. THE Orchestrator `_build_curriculum_graph` method SHALL execute the following steps in order: generate Query_Text, embed the Query_Text, perform Vector_Search, extract Seed_Node IDs, perform Prerequisite_Walk, combine seed and prerequisite node IDs, load full node data with indicators and error patterns, and serialize to JSON.
2. THE Orchestrator SHALL set `ctx.curriculum_graph_json` to the JSON serialization of the retrieved nodes, including node ID, code, title, description, indicators, and error patterns.
3. THE Orchestrator SHALL set `ctx.retrieval_metadata` with keys: `seed_node_ids` (list of UUID strings), `prerequisite_node_ids` (list of UUID strings), `total_nodes_injected` (integer), and `query_text_preview` (first 100 characters of Query_Text).
4. WHEN the combined node set exceeds 25 nodes, THE Orchestrator SHALL log a warning with the node count and student ID.
5. THE Orchestrator SHALL log a structured info message after building the curriculum graph, including seed node count, prerequisite node count, total node count, and student ID.
6. FOR ALL valid retrieval results, serializing the nodes to JSON then deserializing SHALL produce an equivalent list of node dictionaries (round-trip property).

### Requirement 9: ImageAnalysisContext Retrieval Fields

**User Story:** As a developer, I want `ImageAnalysisContext` extended with retrieval metadata and image description fields, so that downstream pipeline steps and audit logs have access to retrieval provenance.

#### Acceptance Criteria

1. THE `ImageAnalysisContext` dataclass SHALL include a `retrieval_metadata` field of type `dict[str, Any]` with a default empty dictionary.
2. THE `ImageAnalysisContext` dataclass SHALL include an `image_description` field of type `str` with a default empty string, storing the Query_Text generated in the image description step.
3. THE new fields SHALL NOT affect existing pipeline steps that do not reference retrieval metadata.

### Requirement 10: EmbeddingService Injection into Orchestrator

**User Story:** As a developer, I want the Orchestrator to receive an EmbeddingService instance via dependency injection, so that the hybrid retrieval pipeline can embed query text without constructing its own service.

#### Acceptance Criteria

1. THE `ImageAnalysisOrchestrator.__init__` method SHALL accept an `embedding_service` parameter and store it as `self._embedding_service`.
2. THE `WorkerService._handle_image_analyze` method SHALL construct or resolve an `EmbeddingService` instance and pass it to the Orchestrator constructor.
3. IF the `embedding_service` parameter is `None`, THEN THE Orchestrator SHALL operate in Fallback_Mode for the `_build_curriculum_graph` step, using the code-ordered SELECT fallback without attempting vector search.

### Requirement 11: ANALYSIS-001 User Template Retrieval Metadata

**User Story:** As a prompt engineer, I want the ANALYSIS-001 user template to include retrieval metadata comments above the prerequisite graph JSON, so that the AI model and human auditors can see which nodes were injected via semantic match vs graph walk.

#### Acceptance Criteria

1. THE ANALYSIS-001 user template SHALL include HTML comment lines above `{{prerequisite_graph_json}}` showing: total nodes injected, seed node codes (semantic match), prerequisite node codes (graph walk), and a query text preview.
2. THE Orchestrator `_render_prompt` step SHALL populate the retrieval metadata template variables from `ctx.retrieval_metadata`.
3. THE ANALYSIS-001 output schema SHALL NOT be modified by this change (output schema changes were completed in Phase 1).

### Requirement 12: Graceful Fallback Behaviour

**User Story:** As a platform engineer, I want the hybrid retrieval pipeline to degrade gracefully to pre-Phase-2 behaviour when embeddings are missing or pgvector is unavailable, so that the image analysis pipeline never crashes due to retrieval infrastructure issues.

#### Acceptance Criteria

1. WHEN embeddings exist for the requested country and subject, THE Orchestrator SHALL use Vector_Search as the primary retrieval path.
2. WHEN no embeddings exist for the requested country and subject, THE Orchestrator SHALL fall back to a code-ordered SELECT of the first 20 indicators joined to their nodes, log a warning with the country and subject, and continue the pipeline.
3. WHEN pgvector raises an `OperationalError` during Vector_Search, THE Orchestrator SHALL catch the error, fall back to the code-ordered SELECT, log an error with the exception details, and continue the pipeline.
4. WHEN the Embedding_Service is unavailable or the embed call fails during query text embedding, THE Orchestrator SHALL use the generic query string `"{subject} {student_grade}"` as the Query_Text and attempt Vector_Search with a pre-computed fallback embedding. IF that also fails, THE Orchestrator SHALL use the code-ordered SELECT fallback.
5. THE Fallback_Mode SHALL produce a `curriculum_graph_json` output that is structurally identical to the normal retrieval output, so that downstream pipeline steps require no conditional logic.
6. THE Orchestrator SHALL set `ctx.retrieval_metadata["fallback_reason"]` to a descriptive string when Fallback_Mode is activated.

### Requirement 13: Embedding Model Consistency Enforcement

**User Story:** As a platform engineer, I want the system to prevent mixing embeddings from different models, so that cosine similarity comparisons are always between vectors from the same embedding space.

#### Acceptance Criteria

1. THE Embedding_Job SHALL check the `embedding_model` column on existing non-null embeddings for the target country and subject before generating new embeddings.
2. IF existing embeddings were generated by a different model than the currently configured Embedding_Service backend, THEN THE Embedding_Job SHALL abort and log an error instructing the operator to run with `force_refresh` to re-embed all indicators.
3. THE Embedding_Service SHALL record the model name (e.g., `"openai-text-embedding-3-small"` or `"minilm-all-MiniLM-L6-v2"`) on each embedded indicator via the `embedding_model` column.
4. THE `_vector_search` method SHALL NOT filter by `embedding_model` — model consistency is enforced at embedding time, not query time.

### Requirement 14: Token Count Reduction Validation

**User Story:** As a platform engineer, I want the hybrid retrieval pipeline to achieve a measurable reduction in curriculum graph token count, so that the accuracy improvement from reduced context noise can be validated.

#### Acceptance Criteria

1. THE Orchestrator SHALL log the token count of `ctx.curriculum_graph_json` after building the curriculum graph, using a tokenizer compatible with the target AI model.
2. WHEN the hybrid retrieval pipeline operates in normal mode (not Fallback_Mode), THE `curriculum_graph_json` token count SHALL be less than 4,000 tokens for a typical country/subject combination.
3. THE logged token count SHALL include the student ID, total node count, and whether Fallback_Mode was activated, for ongoing monitoring.
